# 智研（Zhiyan）创新模块中心

面向本科生科研训练、论文优化与学术规范学习的智能辅助项目。  
本项目由 **腾讯云智能体开发平台工作流 + 前端交互展示网站** 共同构成，目标是将传统“只给结果”的科研智能体，升级为“**可解释、可训练、可迁移、可成长**”的教育型科研辅助系统。

---

## 1. 项目背景

传统科研辅助智能体通常能够完成文献检索、文本润色、基础问答等功能，但在真实使用过程中，仍存在几个明显问题：

- 用户往往只能拿到结果，却不知道“为什么这样改”
- 一次修改很难迁移到下一次写作，缺乏学习闭环
- 风险提示过于分散，不利于快速定位关键问题
- 学术规范教育停留在静态说明层面，缺少互动训练
- 科研训练过程缺少阶段感、成长感与任务感

因此，本项目围绕“**让科研智能体不仅能帮用户完成任务，还能引导用户学会科研表达与规范写作**”这一目标，设计并实现了一套创新模块体系。

---

## 2. 项目定位

本项目适用于以下场景：

- 本科生科研入门训练
- 课程论文与毕业论文写作支持
- 科研写作规范教学
- 论文逻辑审查与风险提示
- 润色结果可解释展示
- 科研任务阶段训练与成长陪伴

项目整体定位为：

> **一个兼具科研辅助、写作训练、风险识别与成长陪伴的教育型智能体系统。**

---

## 3. 核心创新点

### 创新点 1：科研成长任务训练站
将科研过程拆解为任务阶段，以“关卡训练”的形式引导用户完成：

- 风险识别训练
- 文献甄别训练
- 实验设计训练
- 学术表达训练

强调科研训练的“陪伴感”“阶段感”和“成长感”。

---

### 创新点 2：失败案例 / 风险知识联动
围绕科研中常见问题建立风险识别框架，例如：

- 结论过度外推
- 对比实验不足
- 指标定义不清
- 图表描述不完整
- 参考文献支撑不足
- 结构缺项
- 学术表达不严谨
- 数据/数值表述可疑

当前前端已支持风险展示与证据卡片呈现，后续可与平台知识库深度联动，实现失败案例检索与规则预警。

---

### 创新点 3：可解释润色查看器
将普通的“润色结果”升级为“**可学习的润色结果**”。

系统不仅输出润色文本，还同步输出：

- 原句
- 修改后句子
- 修改类型
- 修改原因
- 可迁移规则

用户可以从“看结果”升级为“学方法”。

---

### 创新点 4：科研规范训练场
通过学术规范问答、案例辨析、避坑挑战等形式，把抽象规范转化为交互式训练内容。  
相比传统的说明文档或知识问答，更强调“做题训练 + 即时反馈 + 规则内化”。

---

## 4. 项目架构总览

本项目由两部分组成：

### A. 智能体平台侧（腾讯云智能体开发平台）
负责：
- 用户意图识别
- 工作流调度
- 文献搜索
- 论文润色
- 逻辑审查
- 格式校验
- 图表数据提取
- 结果整合为结构化 JSON

### B. 前端展示侧（本仓库）
负责：
- 报告展示
- 风险可视化
- 可解释润色对照
- 科研规范训练
- 科研成长任务训练
- 消费平台输出的 JSON 并进行交互展示

整体逻辑可以概括为：

```text
用户请求
→ 腾讯云智能体工作流分析处理
→ 输出结构化结果
→ 前端网站导入结果
→ 用户进行可视化查看、训练与交互学习
```markdown

# 8. 本地运行、部署方式、平台联动与项目说明

## 8.1 本地运行

安装依赖

```bash
npm install
```

启动开发环境

```bash
npm run dev
```

构建生产版本

```bash
npm run build
```

## 8.2 GitHub Pages 部署

本项目支持通过 GitHub Actions 自动部署到 GitHub Pages。

典型流程：

1. 将代码推送到 GitHub 仓库
2. 配置 `.github/workflows/deploy.yml`
3. 在仓库 Settings -> Pages 中选择 GitHub Actions
4. 等待 Actions 构建成功
5. 获得公开访问链接

如果项目为 GitHub Pages 仓库子路径部署，需要根据仓库名配置 `vite.config.ts` 的 `base` 路径。

## 8.3 Vercel 部署

若部署到 Vercel，需要注意：

- 构建输出目录应为 `dist`
- `vite.config.ts` 中的 `base` 配置要与部署方式匹配
- 若使用 GitHub Pages 项目路径 `/repo-name/`，则不能直接照搬到 Vercel 根域名部署

推荐做法是根据部署环境动态设置 `base`。

## 8.4 与腾讯云智能体开发平台的联动思路

本项目最关键的价值之一，是将“平台侧智能能力”与“网页侧交互体验”结合起来。

核心联动逻辑如下：

用户输入需求  
→ 腾讯云智能体工作流完成分析  
→ 输出结构化 JSON  
→ 前端网站读取 JSON  
→ 展示结果 / 提供训练 / 进行交互学习

在这个分工中：

- 腾讯云平台负责“智能分析与决策”
- 前端网站负责“结果展示与学习体验”

## 8.5 平台侧推荐工作流设计

对于“全都要”分支，平台应在最后增加一个 JSON 汇总节点，把以下结果整合为统一结构：

- 润色结果
- 逻辑审查结果
- 格式校验结果
- 风险提示结果
- 优先修改建议

最终输出为标准 JSON 文本，供前端展示网站导入。

## 8.6 推荐联动模式

### 模式 A：人工导入 JSON（当前最稳）

平台最终回复用户一段标准 JSON，提示用户：

1. 复制 JSON
2. 保存为 `report.json`
3. 打开报告展示网站
4. 点击“导入报告 JSON”
5. 查看完整报告

**优点：**

- 实现简单
- 稳定
- 比赛演示成功率高

**缺点：**

- 用户操作链路较长

### 模式 B：平台生成带参数链接（可扩展）

平台将 JSON 编码为 URL 参数并拼接到网页链接后：

```
https://your-site/report.html?data=...
```

前端页面自动解码后渲染。

**优点：**

- 自动化程度更高
- 用户体验更顺滑

**缺点：**

- URL 长度有限
- 对大体量 JSON 不够友好

### 模式 C：后端存储 + 页面拉取（长期正式方案）

平台将 JSON 推送到独立后端，由后端生成 `report_id`。  
用户访问：

```
https://your-site/report.html?id=xxxx
```

前端页面通过接口获取报告数据。

**优点：**

- 产品化程度最高
- 可保存历史报告
- 可建立用户成长档案

**缺点：**

- 开发成本更高
- 需要后端与数据库支持

## 8.7 当前推荐的比赛落地方案

对于比赛和短期交付，推荐：

**报告展示网站**

采用 **模式 A：导入 report.json**

**科研成长任务训练站**

由平台输出以下任务参数，再进入网页：

- `topic`
- `keywords`
- `current_stage`
- `task_title`
- `task_desc`

网页根据这些参数加载对应训练任务与关卡。

这样能够保证：

- 平台负责任务发布与智能判断
- 网站负责任务体验与交互训练
